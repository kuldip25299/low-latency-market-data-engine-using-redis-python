# 📚 Glossary

Before diving into the architecture and implementation, let's understand the common terms used throughout this repository.

This project is inspired by how modern trading platforms, low-latency systems, and distributed applications process and distribute market data. If you are new to Redis or trading systems, this glossary will help you understand the terminology used in later chapters.

---

# Market Data

Market data refers to real-time information received from a financial exchange.

Examples include:

- Last traded price
- Bid price
- Ask price
- Volume
- Open price
- High price
- Low price

Example:

| Symbol | Price |
|---------|--------|
| NIFTY | 25,540 |
| TCS | 3,925 |
| RELIANCE | 2,985 |

Every price update is called a **Market Data Update**.

---

# Tick

A **Tick** represents a single market update received from the exchange.

Example

09:15:00

```
NIFTY = 25,540
```

09:15:00.120

```
NIFTY = 25,541
```

09:15:00.250

```
NIFTY = 25,539
```

These are three different ticks.

Modern exchanges generate thousands to millions of ticks every second.

---

# Exchange

An exchange is where financial instruments are traded.

Examples include:

- NSE
- BSE
- NASDAQ
- NYSE

In this project, we simulate an exchange that continuously publishes market prices.

---

# Symbol

A symbol uniquely identifies an instrument.

Examples

```
NIFTY

BANKNIFTY

RELIANCE

TCS

INFY
```

Every symbol continuously receives new market prices.

---

# Producer

A producer generates data.

In our project,

```
Exchange Simulator

↓

Market Data Producer
```

publishes market updates into Redis.

---

# Consumer

A consumer reads data generated by the producer.

Examples

- Dashboard
- Strategy Engine
- Alert Service
- Analytics Engine

One producer can have many consumers.

---

# Publisher

A publisher sends messages to Redis.

Example

```
Publisher

↓

Redis Channel
```

---

# Subscriber

A subscriber listens for new messages.

```
Redis Channel

↓

Dashboard

↓

Strategy

↓

Analytics
```

Whenever a new message arrives, every subscriber immediately receives it.

---

# Cache

A cache is temporary storage used to retrieve frequently accessed data much faster than querying a database.

Instead of repeatedly executing

```sql
SELECT * FROM market_data;
```

we first check Redis.

If the data exists,

Redis immediately returns it.

This significantly reduces database load.

---

# Database Polling

Polling means repeatedly asking the database whether new data is available.

Example

Dashboard

↓

Every second

↓

```
SELECT latest_price
FROM market_data;
```

Even if nothing changed.

Polling wastes:

- CPU
- Network
- Database connections

---

# Push Model

Instead of consumers repeatedly asking for data,

the producer pushes new data immediately.

```
Producer

↓

Redis

↓

Consumers
```

This reduces latency and unnecessary database queries.

Redis Pub/Sub uses the Push Model.

---

# Pull Model

Consumers periodically request data.

```
Consumer

↓

Database

↓

Consumer

↓

Database
```

This is called Pull.

Traditional applications commonly use this model.

---

# Latency

Latency is the time taken for data to travel from one point to another.

Example

Exchange

↓

Redis

↓

Dashboard

If this takes

```
3 milliseconds
```

the latency is

```
3 ms
```

Lower latency is always better for trading systems.

---

# Throughput

Throughput measures how much work a system can process in a given amount of time.

Example

```
500,000 market updates/sec
```

means the throughput is

```
500,000 updates per second.
```

---

# Redis

Redis is an in-memory data store.

Unlike traditional databases, Redis stores most data in RAM, making it extremely fast.

Typical Redis operations take only microseconds.

---

# Redis Hash

A Redis Hash stores related information together.

Example

```
market:NIFTY

price = 25540

volume = 120000

high = 25560

low = 25480
```

Hashes are commonly used for storing the latest market snapshot.

---

# Redis Pub/Sub

Pub/Sub stands for

```
Publish

↓

Subscribe
```

A publisher sends a message once.

Redis immediately delivers it to every subscriber.

```
Producer

↓

Redis

↓

Dashboard

↓

Analytics

↓

Strategy
```

---

# Redis Pipeline

Normally,

100 Redis commands require

100 network trips.

Pipelines allow multiple commands to be sent together.

```
100 Commands

↓

1 Network Round Trip
```

This greatly improves performance.

---

# Latest Price Cache

The latest known market price stored inside Redis.

Example

```
NIFTY

25541
```

Consumers always read this value instead of querying MySQL.

---

# Low-Latency System

A low-latency system is designed to process and deliver information as quickly as possible.

Examples include:

- Trading platforms
- Payment gateways
- Online gaming
- Fraud detection
- Recommendation systems

Reducing latency is often more important than adding new features.

---

# High-Frequency Trading (HFT)

High-Frequency Trading refers to automated trading systems that make decisions within microseconds or milliseconds.

These systems rely on extremely fast market data distribution.

While this repository is **not** an HFT engine, it demonstrates one important component used in such systems: low-latency market data distribution.

---

# Key Takeaways

Before moving to the next chapter, you should understand the following concepts:

- A market exchange continuously publishes price updates.
- Every price update is called a tick.
- A producer generates market data.
- Consumers receive and process that data.
- Polling a database does not scale well.
- Redis provides a fast in-memory layer for data distribution.
- Low latency is critical for modern trading systems.
